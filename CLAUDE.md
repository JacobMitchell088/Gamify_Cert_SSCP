# Gamify_Cert_SSCP

A gamified study site for the **(ISC)² Systems Security Certified Practitioner (SSCP)** certification. Users hit "Play" and rotate through arcade-style mini-games every ~10 questions. Questions are SSCP-domain practice items, generated by free LLMs (OpenRouter `:free` tag) and served from a pre-warmed cache so play never stalls.

---

## 1. Goals & Non-Goals (v1)

**Goals**
- Anonymous play — no login. Streaks/XP in `localStorage`.
- Two playable mini-games at launch, with a plug-in interface so adding more is cheap.
- Question pool that auto-refills in the background; play never blocks on AI.
- Deployable to free-tier hosting end-to-end.

**Non-goals for v1** (revisit later)
- User accounts, persistent leaderboards, social features.
- Mobile-first UI (desktop browser is the v1 target).
- Adaptive difficulty / spaced repetition.
- Spending money on a paid LLM at runtime. (One-time seed gen via a paid/stronger free model is allowed.)

## 2. Architecture

```
┌─────────────────────────┐         ┌──────────────────────────────────┐
│  Frontend (Vercel)      │  HTTPS  │  Backend (Render free)           │
│  React + Vite + Phaser3 │ ──────► │  FastAPI + APScheduler           │
│  Zustand store          │  JSON   │  SQLite (question pool)          │
│                         │         │  OpenRouter racer (httpx async)  │
└─────────────────────────┘         └──────────────────────────────────┘
                                              │
                                              ▼
                                    ┌──────────────────────┐
                                    │ OpenRouter free pool │
                                    │ (race N free models) │
                                    └──────────────────────┘
```

### 2.1 Frontend
- **React 18 + Vite + TypeScript** — fast dev, simple build.
- **Phaser 3** mounted inside a React component (`<GameHost/>`) via a `useRef` canvas. Each mini-game is a `Phaser.Scene` subclass.
- **Zustand** for global state (current run, score, streak, queue of upcoming questions). Lighter than Redux, no boilerplate.
- **TailwindCSS** for menu/HUD UI. Phaser owns the game canvas.
- **API client** is a thin `fetch` wrapper in `src/api/`. No SDK.

### 2.2 Backend
- **FastAPI** (async). Auto OpenAPI docs at `/docs` for free.
- **SQLite** via `sqlmodel` for the question pool. On Render free, disk is ephemeral across deploys — fine, because we always have a seed JSON committed to the repo and re-hydrate on startup.
- **APScheduler** background job polls pool size every ~30s and triggers refill when below threshold.
- **OpenRouter racer**: fires N parallel `httpx` requests to free models; first valid response wins, others are cancelled. Failed/invalid responses are discarded.
- **Validator**: pydantic schema enforces shape (stem, 4 options, exactly one correct, explanation, domain tag). Anything off-spec is rejected.

### 2.3 Hosting (free tier)
- **Frontend → Vercel** (or Cloudflare Pages). Static + CDN, zero cold start.
- **Backend → Render free web service**. 15-min idle sleep is acceptable; frontend shows a "warming up" splash on first request.
- **Secrets** (`OPENROUTER_API_KEY`) live in Render env vars, never in repo.
- If Render becomes painful, **Fly.io** is the fallback (no forced sleep on some configs, but tighter resource limits).

## 3. Question Generation Pipeline

### 3.1 Seed phase (one-time, run locally before first deploy)
- `backend/scripts/generate_seed.py` calls a stronger free model (e.g. `deepseek/deepseek-chat:free` or current best free reasoning model on OpenRouter) in a loop.
- Targets ~50 questions per SSCP domain × 7 domains = ~350 seed questions.
- Output: `backend/app/data/seed_questions.json`, committed to repo.
- **Manual spot-check** a sample for correctness before committing. LLMs hallucinate cert specifics; treat seed as a starting point, not gospel.

### 3.2 Runtime refill
- On backend startup, load seed JSON into SQLite if pool < seed size.
- APScheduler job: every 30s, if `unused_questions < THRESHOLD` (e.g. 50), enqueue refill.
- Refill job: race ~3 free models, validate, dedupe by stem hash, insert.
- Hard cap on monthly OpenRouter calls (env-configurable) so we don't accidentally burn the free quota.

### 3.3 SSCP Domains (tag every question)
1. Security Operations and Administration
2. Access Controls
3. Risk Identification, Monitoring, and Analysis
4. Incident Response and Recovery
5. Cryptography
6. Network and Communications Security
7. Systems and Application Security

## 4. Mini-Game Framework

### 4.1 Contract
Every mini-game implements:
```ts
interface MiniGame {
  key: string;                                // unique id
  scene: typeof Phaser.Scene;                 // Phaser scene class
  presentQuestion(q: Question): void;         // map question into game world
  onAnswer: (correct: boolean) => void;       // callback to host
}
```

### 4.2 Rotation
- Run = 30 questions by default (configurable).
- Every 10 questions, host swaps to the next game in a shuffled rotation.
- Transition screen shows score + next-game preview.

### 4.3 Games at launch
1. **Asteroid Answer Run** — ship flies forward, four answer "portals" approach, player steers into the correct one while dodging debris. Wrong portal = lose shield; correct = score.
2. **Crypto Memory Grid** — flip tiles to match a term (e.g. "AES-256") to a definition. Each successful match counts as one question; mismatch costs time.

### 4.4 Future game ideas (parking lot)
- Tower defense: place a "firewall" turret by picking the right answer.
- Falling-code shooter: shoot the correct answer as choices fall from the top.
- Packet sorter: drag packets to correct VLAN/zone.

## 6. API Surface (v1)

| Method | Path                       | Purpose                                                |
|--------|----------------------------|--------------------------------------------------------|
| GET    | `/health`                  | Liveness check (also wakes Render free instance).      |
| POST   | `/run/start`               | Returns `run_id` + first batch of 10 questions + game. |
| GET    | `/run/{id}/next-batch`     | Returns next 10 questions + next mini-game key.        |
| POST   | `/run/{id}/answer`         | (Optional v1) log an answer for analytics.             |
| GET    | `/pool/stats`              | Debug: unused/total questions, by domain.              |

Questions are returned **without** the `correct_index` field — the backend keeps a server-side map from `question_id → correct_index` and the frontend POSTs back the chosen index for grading. Prevents inspecting network responses.

## 7. Step-by-Step Build Plan

Each step ends in a runnable state. Don't move on until the previous step works.

### Phase 0 — Scaffolding
1. Create conda env `sscp-backend` from `environment.yml` (Python 3.11, FastAPI, httpx, sqlmodel, apscheduler, pydantic, python-dotenv).
2. `backend/app/main.py` → FastAPI app with `/health`. Run `uvicorn app.main:app --reload`.
3. `npm create vite@latest frontend -- --template react-ts` → strip boilerplate, add Tailwind, Zustand, Phaser.
4. Wire a "Play" button that hits `/health` and prints status. Both ends talking → ship phase.

### Phase 1 — Question schema + seed
5. Define `Question` pydantic model (stem, options[4], correct_index, explanation, domain, source).
6. Write `scripts/generate_seed.py` — call OpenRouter once per domain in batches of 10, validate, append to seed JSON. Run locally, spot-check, commit.
7. Build `db.py` and `services/pool.py` — load seed into SQLite on startup; expose `get_batch(n)` and `mark_served(ids)`.
8. Add `GET /pool/stats` for sanity.

### Phase 2 — OpenRouter racer + refill
9. `services/openrouter.py` — async function that takes a list of model IDs, fires parallel requests with `asyncio.wait(..., FIRST_COMPLETED)`, validates, returns first valid Question (or None).
10. APScheduler job (interval = 30s): if `pool.unused_count() < THRESHOLD`, run a refill batch.
11. Monthly call counter in SQLite, hard ceiling from env var.

### Phase 3 — Run lifecycle endpoints
12. `POST /run/start` → creates run row, reserves 10 questions, returns them + `game_key`.
13. `GET /run/{id}/next-batch` → next 10 questions, rotate game.
14. Server-side answer grading endpoint (optional v1).

### Phase 4 — Frontend run shell
15. Zustand `runStore` — current run state, question queue, score, streak, current game.
16. Menu screen → Run screen. Run screen mounts `<GameHost/>` with current `game_key`.
17. Run summary screen at end — score, accuracy, per-domain breakdown.

### Phase 5 — Mini-game #1: Asteroid Answer Run
18. Phaser scene: ship sprite (arrow keys / WASD), four portals as gates, asteroid sprites scrolling.
19. `presentQuestion(q)` paints option text on each portal. Collision with a portal → `onAnswer(idx === correct)`.
20. Smooth handoff back to host (next question or game-swap).

### Phase 6 — Mini-game #2: Crypto Memory Grid
21. Phaser scene with 4×3 tile grid. Each round renders one question as 4 term-tiles + 4 definition-tiles, shuffled.
22. Player flips two tiles; match = correct, mismatch = wrong (or time penalty).

### Phase 7 — Polish + deploy
23. End-of-run XP/streak persisted in `localStorage`.
24. Loading splash that handles Render cold-start gracefully.
25. Deploy backend to Render, frontend to Vercel, wire env vars + CORS.
26. README with setup steps for a new contributor.

### Phase 8+ (later)
- Auth via Supabase, global leaderboards.
- Additional mini-games (see §4.4).
- Spaced repetition / domain-weighted question selection.
- Mobile-friendly layouts.

## 10. Current implementation (drifts from §1–§9 above)

This section captures decisions made after the original plan was written. When the two conflict, this section wins.

### 10.1 Mini-game roster
Three Phaser scenes are wired up; the rotation is configurable in `backend/app/services/games.py`:
- **Tower Defense: Exploit Wave** (`tower_defense`, `TowerDefenseScene.ts`) — the main v1 game. Currently the only entry in `GAME_ROTATION`, so every batch runs TD by default. The frontend menu lets the player override the backend-selected game per run via `runStore.selectedGameKey`.
- **Hacker Boss Duel** (`rpg_boss`, `RpgBossScene.ts`) — selectable from the menu. RPG combat: hero (Red Hood sprite) vs. boss (Mecha-stone Golem). Correct answers parry-counter; wrong answers take a hit. Includes a relic/modifier system (Sharp Edge, Vampire, Iron Stance, Riposte, Second Wind, Streak Rage, Bulwark, Fortune), treasure chambers every 6 fights (every 4 with Fortune), and per-phase boss tinting at 66%/33% HP.
- **Asteroid Answer Run** (`asteroid_answer`, `AsteroidAnswerScene.ts`) — kept as a fallback / variety pick. Was the original v1 prototype before TD took over.

Crypto Memory Grid is **not** implemented; treat it as a future idea from GAMES.md.

### 10.2 No "rounds" — runs are continuous
Original plan rotated games every 10 questions. **That is gone.** A run is 30 questions (`BATCH_SIZE=10 × TOTAL_BATCHES=3` in `backend/app/services/games.py`) and the chosen game persists across batches. TD ends when the core dies or at Q30; RPG ends when the boss or hero dies or at Q30. Per-batch HP/progress resets were a bug — see §10.4.

### 10.3 30-question hard cap is a backend constraint, not a scene rule
RPG can't truly "go until death" past Q30 without lifting `TOTAL_BATCHES` in `services/games.py`. If a player is mid-fight at Q30, the run summary fires regardless. Lifting the cap is a one-line backend change but also affects pool refill cadence — revisit when content volume justifies it.

### 10.4 GameHost must stay mounted between batches
`runStore.advance()` deliberately does **not** set `phase: "loading"` while fetching the next batch. The reason: `<GameHost/>` unmounting destroys the `Phaser.Game` instance, which wipes `this.registry` — and TD/RPG both persist HP and progress in the registry (`td_state_v4`, `rpg_state_v1`). The scene stays on its feedback panel while the next batch arrives, then a `currentQuestion.id` change in the effect at `GameHost.tsx` re-inits the scene with the new question and intact registry state.

If you ever want a true between-batch loading screen, you must move scene state out of `registry` into `runStore` (or `localStorage`) first.

### 10.5 RPG sprite pipeline (Red Hood + Mecha-stone Golem)
Pixel-art spritesheets live in `frontend/public/sprites/`:
- `hood_idle.png` (1344×133) — 12 frames @ 112×133, sourced from row 0 of the Red Hood sheet
- `hood_attack.png` (1344×133) — 12 frames, row 5 (sword slash with arc trail)
- `golem_idle.png` (800×100) — 8 frames @ 100×100, row 1 of the Mecha-stone Golem sheet (crystal pulse)
- `golem_attack.png` (800×100) — 8 frames, row 2 (arm-extension shoot)
- `sky.png`, `moon.png`, `stars.png` — backdrop from EnviroPack02

**Frame dimensions are not guessable.** Source sheets had asymmetric cell pitches (hood 112×133, golem 100×100) that don't match obvious divisors of the sheet dimensions. The detection procedure is: load with Pillow, build per-column alpha-sum arrays, find runs of empty columns, and read the actual cell pitch from the gap-to-gap centroid distances. If you guess (e.g. 96×100 or 125×100), the sliced frames capture parts of two adjacent cells and the character appears to drift sideways across the animation, then snap back at loop restart. There's a working script saved in conversation history — re-derive frame sizes via Pillow before re-extracting if you swap sheets.

The hero sprite faces left in the source sheet, so `drawHero()` applies `setFlipX(true)`. The golem's attack arm extends right in the source, but the boss is positioned to the *right* of the hero on screen, so `drawBoss()` also applies `setFlipX(true)` to make the arm reach left toward the hero. Both spritesheets use `texture.setFilter(NEAREST)` for crisp pixel-art rendering even though Phaser's global config is `antialias: true`.

### 10.6 RPG visual juice
- Hero idle bobs internally in the spritesheet; the shadow gets a separate yoyo scale tween (1.0↔0.85, 750 ms half-cycle) to pulse in time with the bob.
- Golem idle has no built-in motion, so `bossSprite.y` oscillates 0↔-16 over 1.6 s (sine ease), and the boss shadow scales inversely (1.0↔0.7) to fake the height change.
- Telegraph phase shows a 44 px "ATTACK INCOMING" sigil dead-center of the screen, punched in with `Back.easeOut`, that lingers ~1.4 s into the question reading before fading.
- Lunge and recoil tweens target the *container*, not the sprite, so the idle/float tweens on the sprite continue uninterrupted.
- Boss phase tinting is a sprite tint (`setTint`), not a fill color — Phaser tints multiply per channel, so the values are `0xffffff` (full HP), `0xffbbbb` (≤66%), `0xff8888` (≤33%).

### 10.7 Dynamic-height stem and answer cards
Both TD and RPG measure wrapped option text with invisible probe `Text` objects, take the tallest height, then size **all four cards to that height**. This keeps the layout regular while preventing long answers from overflowing. RPG answer cards are clamped to `maxCardH=116` so they fit inside the floor strip (y=600..720); the stem panel is pinned to `bandTop=92` (just under the HUD) instead of vertically centered, to keep the sprites visible.

### 10.8 HUD layout (RPG)
Hero HP bar on the left (x=20, w=480), boss HP bar on the right (x=`GAME_WIDTH−500`, w=480), status text ("Attack N · Streak M") centered between them. Modifier badges row sits below the hero bar at y=56. Bars do not overlap; status text never crosses either bar. If you add more HUD elements, keep them in the gap between bars or below y=82.

### 10.9 Frontend asset paths
- Sprite/background PNGs: `frontend/public/sprites/` → served at `/sprites/...` by Vite. Both dev (`npm run dev`) and build (`npm run build` → `frontend/dist/sprites/`) pick them up automatically.
- Frame dims and animation frame counts are duplicated in `RpgBossScene.preload()` and `RpgBossScene.ensureAnimations()` — they must stay in sync with the actual PNG dimensions.

### 10.10a Deployed prototype (May 2026)
The app is now publicly deployed:
- **Frontend:** Vercel (Vite static build). Vercel Analytics enabled via `<Analytics />` from `@vercel/analytics/react` (NOT `/next` — that subpath is Next.js-only and silently no-ops on Vite).
- **Backend:** Render free web service. 15-min idle sleep is real. Mitigations live on the frontend (see §10.10c).
- **Database:** Supabase Postgres via the Transaction Pooler (PgBouncer in tx mode). Driver is `psycopg[binary]==3.2.10`; `db.py` passes `prepare_threshold=None` + `pool_pre_ping=True` to avoid the "prepared statement \"_pg3_N\" does not exist" error that PgBouncer's tx pooling causes with server-side prepared statements.
- **Python pin:** `backend/.python-version` = `3.11.10`. Without this, Render's autodetect picks 3.14 which has no `pydantic_core` wheel and the build dies in a Rust compile.
- **CORS:** `cors_origins` env on Render must list the Vercel URL with no trailing slash (browser `Origin` headers never carry one). All routes share the same middleware; one entry covers `/health`, `/run/*`, `/feedback`, `/question/{id}/report`.

### 10.10b Question pool persistence
Pool data lives in Supabase Postgres now, not Render's ephemeral disk. Seed JSON (`backend/app/data/seed_questions.json`, ~285 questions at time of writing) is only loaded if the `questions` table is empty (`load_seed_if_empty` in `services/pool.py`). Restarting Render with a populated DB is a no-op for seeding — and logs `"seed skip: pool already has N questions"` so it's obvious what happened.

### 10.10c Cold-start UX (frontend)
The Render free instance sleeps after 15 min; first request can take ~60 s. The frontend mitigates this in two places:
- **Prewarm on landing.** `frontend/src/store/backendStatus.ts` is a tiny Zustand store that polls `/health` every 5 s until the backend answers. `App.tsx` calls `startWarmup()` once on mount, so the cold start runs *while the user reads the menu*, not after they click Play. The Menu shows an amber "Warming up the backend… (~60s)" pill that flips to green "Backend ready" when `/health` succeeds.
- **LoadingScreen timer.** When a Play action does hit a cold backend, the loading view shows a live elapsed counter ("12s / ~60s expected"), a progress bar, and switches to "(taking longer than usual)" past 60 s. No more "is it stuck?" anxiety.

The poll loop is guarded by a module-level `pollTimer` + `pollInFlight` so duplicate `startWarmup()` calls (StrictMode double-mount, navigation back to menu, etc.) don't fan out to N parallel timers.

### 10.10d Feedback button is site-wide; Report Question is in-game only
`FeedbackButton` renders at the top level of `App.tsx` (outside the phase switch) so it's available on menu, overview, loading, summary, review, and in-game. It encodes the current `phase` and `selectedGameKey` into the GitHub issue body as the "Page" metadata — useful for triaging where the user was when they hit submit.

`ReportButton` stays inside the in-game branch only (`if (!currentQuestion) return null` is a defensive secondary guard); it's scoped to a specific question.

### 10.10e /feedback rate limit
`backend/app/routers/feedback.py` runs a small in-memory token bucket: **5 requests per IP per 60 s**. Bucket is a `defaultdict(deque[float])` of `time.monotonic()` timestamps, trimmed and checked on every request. 429 responses carry a `Retry-After` header.

It's safe to keep in-process because Render free runs a single instance — adding a second worker (or moving off Render) would require Redis or similar. Resets on restart, which is fine for spam control (the bucket is purely defensive, not a billing meter).

Real client IP is read from `X-Forwarded-For` first (Render is behind a proxy); falls back to `request.client.host`.

### 10.10f Vercel Analytics events
Custom events are wrapped in `frontend/src/lib/analytics.ts` (`trackEvent(name, props)`) so call sites have a strict event-name union and a single import surface. Tracking failures are swallowed — analytics must never break the app.

Fired events:
- `play_started` — runStore `startRun` after the first batch arrives. Props: `game_key`, `run_id`.
- `play_finished` — fires from three paths, each with a `reason` prop: `completed` (`advance` hits the (currently unreachable) final-batch path), `lost` (`abortRun`, e.g. core/hero death), `quit` (`quitToMenu`). Props: `game_key`, `score`, `total_answered`, `total_correct`.
- `game_selected` — only fires when the key actually changes (no churn from clicking the same card twice).
- `feedback_submitted` — FeedbackButton on 2xx. Props: `category`, `has_contact`, `page`.
- `report_submitted` — ReportButton on 2xx. Props: `question_id`, `had_answered`.

### 10.10g Menu footer / reset progress
Career XP / best streak / runs are stored in `localStorage` under `sscp-progress-v1`. The Menu footer exposes a "Reset progress" link with a confirm dialog → `localStorage.removeItem(...)` → `window.location.reload()`. Useful for QA, and for users who want a clean slate.

### 10.10h Endless Tower Defense (no run-end cap)
`backend/app/services/games.py` no longer defines `TOTAL_BATCHES` — only `BATCH_SIZE = 10`. `_build_batch` always returns `is_final=False`, so the backend will keep handing out batches forever. TD's `buildWaveQueue` removes the old enemy cap and adds super-linear surge past Q20 (`Math.pow(qIndex - 20, 1.35) * 0.6`) plus a late-game HP boost past Q25, so the difficulty curve keeps climbing until the core dies. HUD reads "Wave N" instead of "Q N/30".

RPG still effectively ends at the boss/hero death; it doesn't push the cap because each run is a single fight.

### 10.10i Vault Lockdown shared-graph mechanic (`VaultLockdownScene.ts`)
Replaces the original "6 straight radial paths" design. Each run rolls a fresh random graph that all attackers share:

- **Graph generation** (`generateGraph()` at the bottom of the scene file): vault at center, 6 spawn nodes on the outer ring (angles jittered ±11° off the cardinal sextants), 4–5 random mid-ring nodes, 2–3 inner-ring nodes, and ring 0→1→2→vault edges with occasional same-ring lateral edges. Some spawns also get a 25% shortcut edge directly to an inner node, so a few attackers start inherently closer to the vault. There's a final connectivity sweep that wires any isolated spawn directly to its nearest inner node.
- **Attacker movement** (`nextStepFor()` + `shortestPath()`): every step recomputes a BFS shortest path from the attacker's current node to the vault, treating **edges with ≥1 lock as impassable**. If every route is locked, it falls back to the absolute shortest path and burns a lock on the next edge it crosses (the "lock-busting" branch in `resolveAdvance()`). This keeps the game finite — the player can stall an attacker but can't freeze it forever.
- **Locks live on edges, not paths.** `state.edgeLocks` is `Record<edgeKey, count>` where `edgeKey = "min(a,b)-max(a,b)"`. `MAX_LOCKS_PER_EDGE = 2`. Click an edge during the `placing` phase to drop a lock there.
- **Attackers always sit on nodes.** No mid-edge tweening for resting state — they animate between nodes during a step and snap to the destination on tween complete. Multiple attackers on the same node fan out radially (`attackerSlotOffset()`) so they don't z-fight.
- **Pre-question telegraph.** `state.nextMovers` is pre-computed (`planNextMovers()`) and persisted, so the same two attackers shown pulsing during the answer phase are exactly the ones that resolve in `startAttack()`. `nextMovers[0]` moves on correct OR wrong; `nextMovers[1]` only on wrong. The HUD telegraph line + a banner under the stem spell this out for the player. Movers are re-planned in `advanceQuestion()` after each resolution.
- **Persistence + new-run reset.** `REGISTRY_KEY = "vault_state_v2"` (v1 was the old straight-path version — different key avoids cross-version state pollution). The Phaser.Game is destroyed when `GameHost` unmounts (i.e. on menu→play transitions), which wipes the registry. So every fresh "Play" automatically generates a new graph; mid-run batches reuse the same graph.

If you change `MAX_LOCKS_PER_EDGE`, also update the placing-phase hint text — it quotes the constant literally.

## 11. Dev toggle: reveal correct answer

A single backend flag exposes each question's `correct_index` to the frontend so every option card can render a small "✓ DEV" badge over the right answer for debugging and testing.

**Toggle location:** `backend/app/config.py` → `dev_reveal_answers: bool` (**currently `False`** — flipped off for the deployed prototype / user testing). Flip back to `True` locally if you need the badges back; can also be overridden per-environment via the `DEV_REVEAL_ANSWERS` env var.

**Plumbing:**
- `backend/app/config.py` — the flag.
- `backend/app/models.py` — `QuestionOut.correct_index: int | None`; `Question.to_out(reveal_correct=...)` gates inclusion.
- `backend/app/routers/runs.py` — `_build_batch` reads the flag from settings and passes it through.
- `frontend/src/types.ts` — `Question.correct_index?: number` (optional; absent when flag is off).
- `frontend/src/games/sceneContract.ts` — `makeDevAnswerBadge(scene, question, optionIndex, w, h)` returns a small green badge container when `question.correct_index === optionIndex`, else null. Every scene that renders option cards (`TowerDefenseScene`, `RpgBossScene`, `VaultLockdownScene`, `AsteroidAnswerScene`, `PatchTuesdayScene`) calls this helper at card-build time.

**To remove permanently later:** set the flag to `False` (one line), confirm the frontend `Question.correct_index` field stops being populated, and optionally strip the helper + call sites. The field is optional, so leaving the plumbing in place is harmless.
